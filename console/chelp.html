The General tab - Here are the general configuration items that affect the over all function of Phineas. Note that most of the paths here may be specified relative to the InstallDirectory.

<Phineas>
The following configuration items are found here...
</Phineas>


InstallDirectory - a folder or directory
The InstallDirectory is a full path to where you installed Phineas. It is used to resolve any relative path in the configuration (e.g. a path that doesn't start with a drive letter or leading directory slash).
PartyId - text
Each end point of an ebXML connection is identified by a PartyID. CDC issues these for PHINMS installation. However, it can be any unique identifier you choose, as long as it is acceptible to your communicating endpoint(s). A good alternative is to use the full domain name of the server hosting Phineas (e.g. "my.server.name.and.domain").
Organization - text
Set the Organization to something that readily identifys who is hosting Phineas.
SoapTemplate - a file
The SoapTemplate is the location of the ebXML SOAP template used for composing messages. You should normally not need to change this from the default "templates/soap.xml".
EncryptionTemplate - a file
The EncryptionTemplate is the location of the ebXML encryption template used in the payload part of the messages. You should normally not need to change this from the default "templates/encryption.xml".
CpaTemplate - a file
The CpaTemplate is the location of the ebXML CPA template used for creating new CPAs (Communications Protocol Agreement). You should normally not need to change this from the default "templates/cpa.xml".
CpaDirectory - a folder or directory
CPAs are stored in the folder designed by the CpaDirectory. The CPA (Communications Protocol Agreement) is exchanged between an ebXML sender and receiver and required for communications.
TempDirectory - a folder or directory
Temporary files are placed in the folder designated by the TempDirectory.
LogFile - a file
All general log entries go into the file designated by the LogFile. A new log is started every time Phineas restarts, and the old log is renamed with a timestamp extension.
LogLevel - a drop down list of item selections
The LogLevel determines the type of messages placed into the LogFile, with debug providing the most messages and error the fewest. Setting it to none will suppress logging altogether.
Console Url - text
Phineas can optionally provide a WEB based console for monitoring and configuration. For a stand-alone implementation, the Console Url determine the path used for accessing the console. Note the protocol, host, and port will be added.
Console Root - a folder or directory
The Console Root is the folder where console static content and configuration information is located. This is normally found in the Phineas install path, but can be anywhere on the file system including in a HTTP server root. Note that the dynamic part of the console will not (directly) be rendered by an HTTP server however. Phineas must be running and configured for that.
Console Config - a file
The configuration screen layout of the Phineas console is determined by XML in the Console Config. Depending on the type of Phineas build, parts may be ignored however (for example a SENDER build will ignore RECEIVER configuration items). This file is normally found in the Console Root, but may be located anywhere.
Receiver Url - text
For standalone RECEIVER builds of Phineas, the Receiver Url determines the path used for ebXML requests. For example a typical PHINMS receiver responds to "/receiver/receivefile".
The Server tab - Phineas is typically run as a stand alone program, and thus has it's own build in HTTP(S) server. Settings for that server are configured here.

<Phineas>
<Server>
The following configuration items are found here...
</Server>
</Phineas>


Port - a numeric value
The standalone server non-encrypted HTTP Port may be a standard (e.g. 80) or private (e.g. 8080) value.
NumThreads - a numeric value
The server NumThreads will determine how may simultaneous connection can be made to Phineas. This includes both ebXML and console (browser) requests. Each request gets its own process or "thread". Don't go overboard here. Memory and process related overhead limit the possible improvement in performance.
SSL Port - a numeric value
Set the SSL.Port for encrypted HTTPS connections. It may be a standard (e.g. 443) or private (e.g. 8443) value.
SSL CertFile - a file
Clients connecting to the Phineas server using HTTPS will be presented with the credentials found in the SSL CertFile. This can be in PEM, DER, or PKCS12 formats.
SSL KeyFile - a file
For HTTPS connections, this file holds the private key used by the server. This can be in PEM, DER, or PKCS12 formats, and defaults to the SSL CertFile above.
SSL Password - a password
The SSL Password is use to access encrypted SSL CertFile and/or SSL Keyfile.
SSL AuthFile - a file
The SSL AuthFile is a set of PEM formatted CA (certificate authority) certificates. It is used to authenticate clients. If not given, clients will not be required to authenticate.
The Services tab - When a message is received by Phineas, the service requested determines how (or if) that message will be processed. Services define this processing.

<Phineas>
<Receiver>
<MapInfo>
<Map>
The following configuration items are found here...
</Map>
</MapInfo>
</Receiver>
</Phineas>


Name - text
The Name is used to identify a receiver's service mapping.
Directory - a folder or directory
Received payloads are written to the Directory found in this service.
Queue - a drop down list of item selections
Received messages are recorded in the Queue for this service. See the Queue configurations for more information.
Filter - text
The Filter is a command line program that is expected to read from stdin and write to stdout. The payload may be run through this filter prior to being stored on disk. A typical use would be to data broker an HL7 message.
Service - text
The sender designates the Service in the ebXML which is then matched to the one specified here.
Action - text
The sender designates the Action in the ebXML which is then matched to the one specified here.
Arguments - text
Optional information used by the processing service can be given in the Arguments.
Encryption Type - a drop down list of item selections
Payloads may be encrypted using PEM, DER, or PKCS12 certificates. This specifies the decryption certificate/key. Otherwise payloads are not decrypted.
Encryption Id - text
An Encryption Id specifies optional the DN (distinguished name or Subject) used to match against the sender's encryption certificate.
Encryption Password - a password
The Encryption Password is used to decrypt the certificate/key.
Encryption Unc - text
The certificate or key to use for decryption is specified by the Encryption Unc (Universal Naming Convention). Currently only file UNC's are supported.
The Sender tab - These settings are needed for Phineas to send files and include things like destination, file selections, transport, encryption, and so forth.

<Phineas>
<Sender>
The following configuration items are found here...
</Sender>
</Phineas>


PollInterval - a numeric value
The Phineas sender periodically checks designated folders for files ready to process, and queues for messages ready to send. The PollInterval determines the occurance frequency in seconds of these checks.
MaxThreads - a numeric value
MaxThreads determines how many connections the sender may have active at any one time. Three is a typical maximum value.
CertificateAuthority - a file
The CertificateAuthority is a set of PEM formatted CA (certificate authority) certificates. It is used to authenticate receivers for HTTPS connections. If not given, receivers will not be required to authenticate.
MaxRetry - a numeric value
If a send attempt fails (for example, if the host is not responding), the sender will normally retry. The MaxRetry determines the number of attempt that will be made.
DelayRetry - a numeric value
Before a sender retries sending a message, it will wait for the receiver/network to correct whatever condition may have caused the failure. The DelayRetry determines the intial wait in seconds. Each subsequent retry will double the wait interval for an exponential retry back-off.
The Maps tab - A sender's map associates files with a Route and Queue. When those files get selected for transport they will be recorded in the Queue and sent to the Route's destination.

<Phineas>
<Sender>
<MapInfo>
<Map>
The following configuration items are found here...
</Map>
</MapInfo>
</Sender>
</Phineas>


Name - text
The Name of the sender map associated with a file folder.
Queue - a drop down list of item selections
The name of the Queue used to record files processed from a folder map.
Processor - a drop down list of item selections
The type of Processor used for this folder. Currently only ebxml is supported. Future versions may include other protocols like MLLP.
Folder - a folder or directory
The Folder is periodically checked for files to be added to a queue (and subsequently processed or sent). This can include wild card designations to restrict which files get selected (for example, "data/outgoing/*.txt").
Processed - a folder or directory
The Processed folder gets files after they have been processed by the folder poller.
Acknowledged - a folder or directory
After the queue processors is finished it will add a status message from the remote receiver to a file in the Acknowledged folder.
Route - a drop down list of item selections
The Route designates the payload destination for a selected folder mapping.
Service - text
The Service (and Action) identifies how the receiver should handle the incoming request (payload).
Action - text
The Action (and Service) identifies how the receiver should handle the incoming request (payload).
Arguments - text
The send can pass optional Arguments to the receiver to further affect processing of the request.
Encryption Type - a drop down list of item selections
If payload encryption is desired, the Encryption Type determines the method. Currently on certificate encryption is supported.
Encryption Id - text
An Encryption Id specifies optional the DN (distinguished name or Subject) identified to the receiver in the encryption payload.
Encryption Password - a password
The Encryption Password is used to decrypt the certificate.
Encryption Unc - text
The certificate to use for encryption is specified by the Encryption Unc (Universal Naming Convention). Currently only file based names are supported. This can be in PEM, DER, or PKCS12 formats.
The Routes tab - A Route is used by the sender to select a physical end point for the transport. That includes things like the domain name, path, ports, and protocol. It also determines the CPA required by the receiver.

<Phineas>
<Sender>
<RouteInfo>
<Route>
The following configuration items are found here...
</Route>
</RouteInfo>
</Sender>
</Phineas>


Name - text
The Name identifies a sender's route.
PartyId - text
The PartyId identifies the receiver for a route. CDC issues these for PHINMS installation. However, it can be any unique identifier designated by the receiver. See the General PartyID for details.
Cpa - text
The Cpa identifies the certificate protocol agreement file used for the route connection. It is kept in the CpaDirectory. The naming convention is the receiver and sender PartyId separated by a period.
Host - text
The Host is the domain or IP designation for the receiver (e.g. "some.remote.receiver.gov").
Path - text
The Path is the URL path expected by the receiver (typically "/receiver/receivefile" for PHINMS hosts).
Port - a numeric value
The Port is the port used by the receiver (typically 80 for non-encryption, or 443 for encrypted connections).
Protocol - a drop down list of item selections
The Protocol determines whether SSL encryption and optional authentication is selected for the connection.
Timeout - a numeric value
After the sender transmits a payload request, it waits for a response. If it waits longer than the Timeout in seconds, it assumes there is some problem with the receiver or networks and optionally retries the send.
Retry - a numeric value
After a Timeout, an attempt may be made to resend the message. The Retry determines how many times that attempt will be made.
Authentication Type - a drop down list of item selections
If the sender must authenticate itself to the receiver for an SSL connection, the method is specified by the Authentication Type. Currently only certificate authentication is supported.
Authentication Id - text
An Authentication Id specifies the DN (distinguished name or Subject) identified in the encryption payload envelop. It is only needed to select a specific certificate from a store, and is normally derived from the certificate if left blank. This will be matched against the receiver's decryption certificate.
Authentication Password - a password
The Authentication Password is used to optinally decrypt the Authetication Unc.
Authentication Unc - text
The certificate store used for authentication is specified by the Authentication Unc (Universal Naming Convention). Currently only file based names are supported. This can be in PEM, DER, or PKCS12 formats.
Queue - a drop down list of item selections
The Queue specifies where an audit record of the sender's transport is recorded. For routes it is only needed for Ping requests.
Name - a submit button
Click Export.CPA to create a communication protocol agreement file for this route.

The Q Types tab - The Queue Type identifies the fields of Queue, although the physical implementation may include additional fields and/or have order differences. Queue Types should begin with a "RECORDID" field. EbXml queues are expected to have a specific set of fields present, so you should NOT delete any found there. The EbXml defaults are fully compatible with existing PHINMS databases.

<Phineas>
<QueueInfo>
<Type>
The following configuration items are found here...
</Type>
</QueueInfo>
</Phineas>


Name - text
The Name identifing the queue type.
Field - text
One or more Field names for the queue. Note that all queues are expected to begin with a RecordId. The Add Field button can be used to add additional fields. Blank fields are removed.
The Connections tab - A Phineas Queue is associated with a Connection that determines the physical implementation of a queue. This is typically a database connection, but Phineas also offers a light weight file based connection.

<Phineas>
<QueueInfo>
<Connection>
The following configuration items are found here...
</Connection>
</QueueInfo>
</Phineas>


Name - text
The Name used to identify a queue connection.
Type - a drop down list of item selections
The type of connection. Use "file" for file based or "odbc" for an ODBC connection.
Id - text
The Id is optionally used to authenticate the queue connection.
Password - a password
The Password is optionally used to authenticate the queue connection.
Unc - text
A queue connection is accessed by a Unc (Universal Naming Convention) path. The nature of the path is determined by the connection type (for example ODBC conventions).
The Load/Save/Update tab - The Phineas configuration is loaded from a disk file when the program starts. That is the RUNNING configuration. The configuration being editted (here) may be loaded or saved from that RUNNING configuration, or elsewhere. Similarly changes made in the web configuration GUI may be updated (in memory), or discarded by simply navigating away in the console.
Configuration File Name - a file
The configuration for Phineas is read or written to the Configuration File Name.
Save Configuration - a submit button
Click Save Configuration to save the current configuration in edit to disk.
Load Configuration - a submit button
Click Load Configuration to Load the current configuration in edit from disk.
Update Configuration - a submit button
Click Update Configuration to keep changes to the current configuration in edit. Otherwise changes may be discarded after you navigate away from the configuration screen.
